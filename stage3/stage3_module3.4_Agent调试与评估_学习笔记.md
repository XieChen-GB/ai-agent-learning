# Stage 3 · Module 3.4 — Agent 调试与评估 学习笔记

## 本节目标

1. 用 LangSmith 追踪 Agent 每一步的输入/输出
2. 定义评估指标：工具是否调用、搜索是否真的命中、回答是否相关、耗时
3. 通过一次真实的调试过程，识别 Agent 常见失败模式

对应代码：`06_agent_evaluation.py`

---

## 一、为什么 Agent 调试比普通程序难

普通程序出错会报异常，定位到具体行。Agent 不一样：

```
用户输入 → Prompt 构造 → LLM 推理（黑箱）→ 工具选择 → 工具执行 → 结果回传 → 最终回答
```

任何一层都可能出问题，而且**不一定报错**——Agent 可能安静地给出一个错误或无意义的答案，没有任何异常信息。这叫"静默失败"（silent failure），是 Agent 调试最棘手的地方。

LangSmith 的作用：像飞机黑匣子一样，把每一层的输入输出都记录下来，方便事后回放分析。

---

## 二、评估指标设计

`run_agent` 函数里记录的指标：

| 指标 | 含义 | 为什么需要 |
|------|------|-----------|
| `tool_called` | 模型是否决定调用工具 | 判断模型有没有意识到需要查数据 |
| `search_found_results` | 工具搜索是否真的找到了记录 | 判断检索本身有没有效，不能只看"工具被调用了" |
| `answer_relevant` | 最终回答是否包含预期关键词 | 粗略判断回答相关性 |
| `elapsed_seconds` | 总耗时 | 性能监控 |

**重要教训：`answer_relevant` 这个指标本身有缺陷。**

它只做字符串匹配，不懂语义。曾经出现过这种情况：模型回答"我没有找到关于销售额增长的信息"，这句话里包含"销售额"三个字，导致 `answer_relevant` 被误判为 `True`，但实际上回答内容是空的。

**结论：简单的关键词匹配评估法，在 Agent 说"找不到"的情况下会产生假阳性。** 更可靠的评估应结合 `search_found_results` 一起看，不能只看一个指标。

## 三、本节调试过程全记录

这是本节真正的核心产出——一次完整的"发现问题 → 修复 → 引发新问题 → 再修复"的循环。

### 第一轮：发现关键词不匹配问题

**现象：** 三个测试问题，工具都被调用了，但全部返回"没有找到相关记录"。

**根因排查：**
- 检查 `meeting_data.json` 原文，确认"销售额"、"宕机"、"融资"这些词确实存在于数据里
- 检查模型实际生成的搜索关键词：`"销售额增长"`、`"技术团队问题"`、`"融资计划"`
- 真相：工具用的是**精确子串匹配**（`keyword in r`），要求关键词完整出现在记录里。模型生成的关键词是组合长词，跟原文不完全一致，导致匹配失败

**结论：这是典型的"工具检索逻辑跟不上模型自然语言生成习惯"的失败模式。**

---

### 第二轮：第一次修复（改 docstring）——引入新问题

**修复方式：** 在 `search_meeting_data` 的 docstring 里加入强约束：

```python
"""
keyword: 要搜索的关键词，必须是 2-4 个字的核心词，不要用完整短语。
好例子："销售额"、"融资"、"宕机"
坏例子："销售额增长情况"、"技术团队遇到的问题"
"""
```

**结果：** 对"技术团队"和"融资"两个问题有效，但"销售额"问题出现了新故障——模型完全没有调用工具，而是反过来质问用户"你的关键词太长了，请提供 2-4 字的核心词"。

**根因：** 模型把 docstring 里的约束误解成了"对用户的要求"，而不是"自己提炼关键词时要遵守的规则"，把任务责任推给了用户。

**教训：约束写得越死，模型的行为越不可预测，有时会"读过头"，产生意料之外的新失败模式。调试 Agent 不是一次性修复，是持续观察—调整—再观察的循环。**

---

### 第三轮：第二次修复（fuzzy_match 模糊匹配）

放弃"用 Prompt 去控制模型生成什么样的关键词"的思路，改为在工具内部放宽匹配逻辑，不再依赖模型听不听话：

```python
def fuzzy_match(keyword: str, text: str) -> bool:
    # 关键词很短（2字以内）时，用原来的子串匹配
    if len(keyword) <= 2:
        return keyword in text
    # 关键词较长时，只要任意连续2字出现在text里就算命中
    for i in range(len(keyword) - 1):
        if keyword[i:i+2] in text:
            return True
    return False
```

同时把 docstring 改回简单版本，不再对模型做强约束：

```python
"""
keyword: 要搜索的关键词，例如"销售额"、"融资"
"""
```

**结果：三条测试全部 `search_found_results: True`。** 不管模型生成长词还是短词，工具都能兜底找到结果。

## 四、仍然存在、留待下次处理的问题

"技术团队最近有什么问题？"这条，工具搜到了相关记录（包含"宕机"的那条），但最终 `answer_relevant` 仍是 `False`。

观察模型的实际回答，引用的是"技术总监离职""安全渗透测试"等内容，没有提到"宕机"那条记录。

**可能原因（待下次验证）：**
1. `fuzzy_match` 过于宽松，一次搜索命中了多条不完全相关的记录，模型在生成最终回答时选择性地引用了其中几条，"宕机"那条被忽略了——这是"上下文丢失/信息选择偏差"，属于另一种失败模式，不是检索失败
2. 也可能是模型本身在多条候选信息里做摘要时的随机性导致

**下一步方向（已记录到 3.1 RAG 待整合清单）：**
用向量检索（3.1 已学的 ChromaDB + embedding）替代当前的字符串模糊匹配，从语义层面解决关键词不对齐问题，比 `fuzzy_match` 这种字符层面的折中方案更根本。

---

## 五、本节关键认知总结

1. Agent 的错误经常是**静默的**，必须依赖 LangSmith 这类追踪系统才能发现问题出在哪一层
2. 评估指标本身也需要被审视，单一指标（如关键词匹配）容易产生误判，需要多指标交叉验证
3. 修 Prompt/docstring 解决一个问题，往往会引入新的、意料之外的行为偏差——这是 Agent 工程和传统编程最大的不同点：传统代码改了就是改了，Prompt 改了之后模型的反应有不确定性
4. 工具层面的兜底（如模糊匹配）比单纯依赖"模型听话"更可靠，但也有上限——真正的根治方案是语义检索（RAG），留给 3.1 整合阶段处理
