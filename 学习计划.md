**什么是数据结构？**
数据结构是组织和存储数据的方式。Python 有三种最常用的内置数据结构：

- **集合（Set）**：无重复元素的集合，用于去重和快速查找

**什么是错误处理（try/except）？**
程序运行时不可避免会出错（文件不存在、网络断了、数据格式错误等）。错误处理让程序在出错时不直接崩溃，而是优雅地处理异常并继续运行。Agent 必须有完善的错误处理，否则一个工具失败整个流程就停了。
 
```python
import json
 
# 读写 JSON 文件（存储 Agent 的记忆/配置）
def save_notes(notes, filepath="notes.json"):

        json.dump(notes, f, ensure_ascii=False, indent=2)
 
def load_notes(filepath="notes.json"):
    try:
        with open(filepath, "r", encoding="utf-8") as f:
            return json.load(f)
    except FileNotFoundError:
        return []   # 文件不存在时返回空列表
    except json.JSONDecodeError:
        print("文件格式错误")
        return []
 
# 错误处理（Agent 必须健壮）
def safe_divide(a, b):
    try:
        return a / b
    except ZeroDivisionError:
        return None
    except TypeError as e:
        raise ValueError(f"参数类型错误: {e}")
```

### 1.6 第六周：异步编程（AI Agent 必学）
 
**目标：** 理解 async/await，这是调用 AI API 的标准方式
 
**什么是异步编程？**
想象你在餐厅点了三道菜。同步方式是：厨师做完第一道才开始做第二道，客人等很久。异步方式是：三道菜同时开始做，哪个先好上哪个，总时间大幅缩短。
 
程序里也一样：Agent 同时调用"搜索网页""读取文件""查询数据库"三个工具时，异步可以让它们并行执行，而不是排队等待。调用 AI API 时由于网络延迟较大，异步尤其重要。
 
**关键词解释：**
- `async def`：定义一个异步函数
- `await`：等待某个异步操作完成（等待期间 CPU 可以去做其他事）
- `asyncio.gather()`：并行运行多个异步任务，全部完成后统一返回结果
```python
import asyncio
 
# 为什么需要异步？
# Agent 同时调用多个工具时，不应该一个等一个完成
# 异步让它们并行执行，速度提升几倍
 
async def call_tool(tool_name, delay):
    print(f"开始调用 {tool_name}")
    await asyncio.sleep(delay)       # 模拟 API 调用延迟
    print(f"{tool_name} 完成")
    return f"{tool_name} 的结果"
 
async def main():
    # 并行调用多个工具（比串行快得多）
    results = await asyncio.gather(
        call_tool("搜索网页", 2),
        call_tool("读取文件", 1),
        call_tool("发送邮件", 3),
    )
    print(results)
 
asyncio.run(main())
```

## 阶段 1.5：FastAPI 基础（3～5 天）
 
> Agent 需要对外提供接口、接收 Webhook，FastAPI 是 Python 世界的标配，早学早用
 
**什么是 FastAPI？**
FastAPI 是一个用 Python 编写的现代 Web 框架，由哥伦比亚开发者 Sebastián Ramírez 于 2018 年发布。它让你用很少的代码就能搭建一个可以接收 HTTP 请求、返回 JSON 数据的 API 服务。"Fast"有两层含义：开发速度快 + 运行性能快。
 
**什么是 API？**
API（Application Programming Interface，应用程序接口）是程序之间通信的"插头和插座"。就像电器的插头只要符合规格就能插进任何插座，API 定义了程序之间交换数据的格式和方式。当你的 Agent 需要接收来自外部的消息（比如新邮件到达的通知），或者对外提供服务（比如让网页调用你的 Agent），都需要通过 API。
 
**什么是 Webhook？**
Webhook 是一种"主动推送"机制。普通 API 是你去问（"有新消息吗？"），Webhook 是对方主动通知你（"有新消息了！"）。比如：邮件系统收到新邮件后，自动发一个请求到你 Agent 的 URL，触发处理流程。
 
**什么是 JSON 格式的请求和响应？**
API 通信的标准格式。请求是你发给服务器的数据，响应是服务器返回给你的数据，都用 JSON 格式打包。
 
### 目标
能写出一个可以接收请求、返回 JSON 的 API 服务
 
### 核心内容
 
```python
from fastapi import FastAPI
from pydantic import BaseModel
 
app = FastAPI()
 
# GET 请求：查询数据
@app.get("/agent/status")
def get_status():
    return {"status": "running", "model": "qwen2.5:7b"}
 
# POST 请求：接收用户输入，返回 AI 处理结果
class ChatRequest(BaseModel):
    message: str
    session_id: str
 
@app.post("/agent/chat")
async def chat(request: ChatRequest):
    # 后续接入 LLM
    return {"reply": f"收到：{request.message}", "session": request.session_id}
```

## 阶段 2：AI 入门基础（4～6 周）
 
### 目标
能写出一个可以调用工具的简单 Agent
 
**什么是 AI Agent？**
Agent 是一个能够自主完成任务的 AI 程序。和普通 AI 聊天不同，Agent 有以下能力：
- **感知**：接收输入（文字、文件、网页内容等）
- **思考**：用 LLM 判断下一步该做什么
- **行动**：调用工具（搜索网页、发邮件、读文件等）
- **记忆**：记住对话历史和执行进度
一个简单的比喻：普通 AI 是"顾问"，你问它一个问题它给你一个建议；Agent 是"员工"，你给它一个目标，它自己想办法完成，包括调用各种工具、处理中途的问题。
 
**什么是 LLM（大语言模型）？**
LLM（Large Language Model）是驱动 Agent 的"大脑"。它是在海量文本上训练出来的神经网络，能理解和生成自然语言。常见的 LLM 有：Claude（Anthropic）、GPT-4（OpenAI）、Gemini（Google）、Qwen（Alibaba）。LLM 本身不能联网、不能操作文件，Agent 的作用就是给 LLM 配上"手脚"（工具）。

### 2.1 Prompt Engineering（1 周）
 
> 越早学越关键，决定 Agent 质量的上限
 
**什么是 Prompt Engineering（提示词工程）？**
Prompt 就是你发给 AI 模型的输入文字。Prompt Engineering 是一门研究"如何写出更好的输入，让模型输出更准确结果"的技术。同样的问题，不同的写法可能让模型的回答质量差异巨大。这不是玄学，而是有系统方法论的工程实践。
 
**四个核心技巧：**
- **System Prompt（系统提示）**：在对话开始前设定模型的角色和行为规范，比如"你是一个严格按 JSON 格式输出的数据提取助手"
- **Few-shot（少样本示例）**：给模型展示 2～3 个输入/输出的例子，模型会按例子的风格回答
- **Chain of Thought（思维链）**：在 Prompt 里加上"请一步一步思考"，让模型先推理再给答案，准确率显著提升
- **输出格式控制**：明确要求模型只输出 JSON，不要多余的文字，方便程序解析
**练习：** 用 Ollama，写一个 Prompt，让模型把任意一段中文会议记录提取成结构化 JSON
 
### 2.2 API 与 Function Calling（1～2 周）
 
> 用 Ollama 免费练，语法和 OpenAI / Anthropic 基本一致
 
**什么是 Function Calling（函数调用）？**
这是 LLM 最重要的能力之一。你提前告诉模型"你可以使用这些工具（函数）"，当用户提问时，模型会自动判断：
1. 这个问题需要调用哪个工具？
2. 需要传什么参数？
然后由你的代码实际执行工具，把结果还给模型，模型再根据结果给出最终回答。
 
**举例：**
用户问"东京今天天气怎么样？"→ 模型判断需要调用 `get_weather` 工具，参数是 `city="东京"` → 你的代码调用天气 API 拿到数据 → 模型用这个数据回答用户
 
这就是 Agent 能"做事"而不只是"说话"的核心机制。
 
**核心概念：**
- `messages` 数组结构（role: system / user / assistant / tool）
- Function Calling：告诉模型"你可以用这些工具"
- Tool Result：把工具执行结果还给模型
```python
# Ollama 调用示例（免费）
import ollama
 
tools = [{
    "type": "function",
    "function": {
        "name": "get_weather",
        "description": "获取某城市天气",
        "parameters": {
            "type": "object",
            "properties": {
                "city": {"type": "string", "description": "城市名"}
            }
        }
    }
}]
 
response = ollama.chat(
    model="qwen2.5:7b",
    messages=[{"role": "user", "content": "东京今天天气怎么样？"}],
    tools=tools
)
```
 
**练习：** 实现一个"工具调用闭环"——模型判断需要工具 → 你执行工具 → 把结果返回模型 → 模型给出最终回复
 
### 2.3 状态管理基础（3～5 天）
 
> Agent 最大难点之一不是 Prompt，而是 **State（状态）**——Agent 当前在哪一步、做了什么、失败了怎么恢复
 
**为什么重要：** 没有状态管理的 Agent，一旦中途出错就完全不知道恢复到哪里，也无法处理多轮复杂任务
 
**核心概念：**
 
```python
# Agent 状态示例——用字典追踪执行进度
agent_state = {
    "current_step": "extract_action_items",   # 当前执行到哪
    "completed_tools": ["transcribe", "summarize"],  # 已完成的工具
    "user_confirmed": False,                  # 用户是否确认
    "retry_count": 0,                         # 失败重试次数
    "last_error": None,                       # 上次错误信息
    "resume_from": None,                      # 失败后从哪恢复
}
 
def update_state(state, step, result):
    state["completed_tools"].append(step)
    state["current_step"] = result
    return state
```
 
**有限状态机（FSM）概念——了解即可：**
 
```
[等待输入] → [处理中] → [等待用户确认] → [执行工具] → [完成]
                ↓                               ↓
            [出错重试]  ←─────────────────  [工具失败]
```
 
**学习清单：**
- [ ] 理解 State 的概念：Agent 的"记忆"不只是对话历史，还有执行进度
- [ ] 用字典实现一个简单的状态追踪器
- [ ] 了解有限状态机（FSM）是什么，知道 LangGraph 就是基于这个思想
**练习：** 给上一节的"工具调用闭环"加上状态追踪——记录每个工具是否执行成功，失败时能从断点重试

### 3.1 向量数据库 + RAG（2 周）
 
> 先理解原理，再学框架
 
**什么是向量（Embedding）？**
向量是把文字转换成一串数字的技术。比如"苹果"这个词可以被转换成 `[0.2, 0.8, -0.3, ...]` 这样一串数字。神奇的是，意思相近的词转换出来的数字也会很接近。这让计算机可以通过数学计算来理解语义相似度。
 
**什么是向量数据库？**
专门用来存储和检索向量的数据库。普通数据库用关键词匹配（完全相同才能找到），向量数据库用语义相似度匹配（意思接近就能找到）。比如搜索"业绩"，能找到包含"销售额""营收"的文档。
 
**什么是 RAG（检索增强生成）？**
RAG（Retrieval-Augmented Generation）解决了 LLM 的一个核心问题：LLM 只知道训练时学过的内容，不知道你的私有文档。RAG 的做法是：先从你的文档库里检索出相关内容，再把这些内容和问题一起发给 LLM，让它基于你的文档来回答。
 
**什么是 ChromaDB？**
ChromaDB 是一个开源的本地向量数据库，由美国公司 Chroma 开发（2022年）。可以直接在你的 MacBook 上运行，不需要服务器，数据存在本地。是学习阶段最推荐的向量库，安装简单、上手快、完全免费。
 
**RAG 工作流程：**
1. 把文档切块 → 转成向量（Embedding）→ 存入向量库
2. 用户提问 → 问题也转成向量 → 找最相似的文档块
3. 把文档块 + 问题一起发给 LLM → 得到基于文档的回答
**工具推荐（免费）：**
- 向量库：`ChromaDB`（本地，无需服务器）
- Embedding 模型：Ollama 内置（`nomic-embed-text`）
```python
import chromadb
from ollama import embeddings
 
# 创建本地向量库
client = chromadb.Client()
collection = client.create_collection("meeting_notes")
 
# 存入文档
text = "Q3 销售额同比增长 23%，主要来自亚太区"
embed = embeddings(model="nomic-embed-text", prompt=text)["embedding"]
collection.add(embeddings=[embed], documents=[text], ids=["doc1"])
```
 
**练习：** 把 10 篇会议纪要存入 ChromaDB，实现"问任何问题，从纪要中找答案"

**什么是 LangChain？**
LangChain 是 2022 年推出的 Python 框架，目标是简化 LLM 应用开发。它把常见操作（调用模型、存记忆、检索文档）封装成标准组件，像乐高积木一样拼接。但因为封装层太厚、学习曲线陡、版本变化频繁，现在行业趋势是只用它的调试工具 LangSmith，工作流部分改用 LangGraph。

**需要了解的概念（知道是什么就够）：**

### 3.3 MCP 协议（1 周）
 
> **你有 Claude Desktop，可以直接实践！**
 
**什么是 MCP（Model Context Protocol）？**
MCP 是 Anthropic 于 2024 年发布的开放协议，目标是让 AI 模型连接外部工具的方式标准化。类比：就像 USB 接口让各种设备都能连接电脑，MCP 让各种工具（文件系统、数据库、API、浏览器等）都能以统一方式连接 AI 模型。
 
在 MCP 出现之前，每个 AI 应用都要自己写连接各种工具的代码，重复劳动且不兼容。有了 MCP，工具开发者只需写一次 MCP Server，所有支持 MCP 的 AI（Claude、Cursor 等）都能直接使用。
 
**你已经在用 MCP 了：** Claude Desktop 内置了 MCP 客户端，你启用"文件系统"功能时，背后就是 MCP 在工作。
 
**MCP 的两个角色：**
- **MCP Server**：提供工具能力的程序（比如"可以读写本地文件"）
- **MCP Client**：使用工具的 AI 应用（比如 Claude Desktop）
**你现在就能做：**
1. 在 Claude Desktop 中启用已有 MCP 服务（文件系统、浏览器等）
2. 观察 Claude 是如何决定调用哪个工具的
3. 用 Python 写一个自定义 MCP Server，暴露给 Claude Desktop 调用

### 4.1 多 Agent 系统（3 週）
 
**什么是多 Agent 系统？**
单个 Agent 处理复杂任务时容易出错，就像一个人又要做计划又要执行又要检查，容易顾此失彼。多 Agent 系统把任务分配给多个专门的 Agent，各司其职，协同完成。
 
**什么是 LangGraph？**
LangGraph 是 LangChain 团队于 2024 年推出的框架，专门用于构建多步骤、有状态的 Agent 工作流。它把 Agent 的执行过程可视化为一张"图"（Graph），每个节点是一个处理步骤，边是流转条件。本质上就是阶段 2.3 里学的状态机的工程化实现。
 
**什么是 AutoGen？**
AutoGen 是微软研究院开发的多 Agent 框架（2023年），特点是让多个 AI Agent 像人类团队一样通过对话协作解决问题。比如一个 Agent 写代码，另一个 Agent 审查代码，来回讨论直到达成共识。
 
**核心模式：**
- **Orchestrator + Worker**：一个主 Agent 分配任务，多个子 Agent 执行
- **Pipeline**：每个 Agent 处理一个环节，串联成流程
- **反思模式**：一个 Agent 生成，另一个 Agent 检查和改进
**框架：**
- `LangGraph`：基于状态机思想的 Agent 工作流（**重点学，是行业主流**）
- `AutoGen`：微软出品，多 Agent 对话框架
> **LangGraph 和阶段 2.3 的状态管理直接对应：** LangGraph 本质上就是把你手写的状态字典，变成可视化、可调试的状态机图。学过状态管理之后再接触 LangGraph 会非常顺畅
 
### 4.2 错误处理与健壮性（1 周）

# Agent 在真实环境中必须处理失败
async def call_tool_with_retry(tool, args, max_retries=3):
    for attempt in range(max_retries):
        try:
            result = await tool(**args)
            return result
        except Exception as e:
            if attempt == max_retries - 1:

| 生产环境 | **AWS Lambda** | 按实际调用次数计费，适合流量不稳定的场景 | 按需 |
 
**什么是 Railway / Render？**
两者都是面向开发者的云平台，特点是"连接你的 GitHub 仓库，推送代码自动部署"，操作极简。免费层对个人项目够用。Railway 界面更现代，Render 免费层额度更大。
 
> **推荐路径：** 本机开发调试 → Railway 免费层试部署 → 按需升级到 Fly.io 或 AWS
 
**成本控制技巧：**
- 缓存重复查询（同样的问题不重复调用 API）
- 用小模型（Haiku）做分类和路由，大模型只做最终生成
- 设置每日 Token 上限，防止意外超支
- M2 本地模型能处理的任务，尽量不调用付费 API

## 🚀 贯穿全程的实践项目：自动会议纪要 Agent
 
> 覆盖 Agent 所有核心能力，建议从阶段 1 结束后开始，边学边做
 
### 完整流程
 
```
用户上传会议录音（.mp3 / .mp4）
        ↓

        ↓

        ↓

        ↓

        ↓

        ↓
同步 Action Items 到日历 / Todo 系统
```
